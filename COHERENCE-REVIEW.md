# Coherence & Completeness Review — Self-Healing Ops (6 component specs, reframed system)

Read as one system: `LOOP-A-SPEC.md`, `LOOP-B-SPEC.md`, `VERIFICATION-GATE.md`, `INCIDENT-MEMORY.md`, `TRUST-CONTROLLER.md`, `SECURITY-THREATMODEL.md`, plus the two referenced-but-unwritten keystones `ORCHESTRATION.md` and `HITL-APPROVAL.md`, and the pending `ARCHITECTURE-REFRAMED.md`. Findings are prioritized: BLOCKERS (a build would produce two components that do not interoperate) first, then MAJOR (a decision/attack is unowned or double-owned), then MINOR (nits the keystone should still note). Each names exactly what the keystone documents must reconcile.

## BLOCKERS — interface mismatches that break interop as written

**1. `OutcomeEvent` field name mismatch: `autoActionId` vs `actionId`.** INCIDENT-MEMORY §7.1 emits `OutcomeEvent = { autoActionId, kind, at }` and says it is the "EXACT shape the controller's `ingestOutcome` expects — no renames." TRUST-CONTROLLER's actual interface (§Interfaces, and `outcomeWeight(actionId)`) defines `OutcomeEvent = { actionId, kind, at }`. These are different field names on the one contract that couples the two components. As written, `ingestOutcome(ev)` reads `ev.actionId` = `undefined`. **Keystone must pin one name** (recommend `actionId`, since `assisted_action`s are also actions, not auto-actions — see #3) and correct the other spec.

**2. Maturation window contradiction: `W_confirm` (14d) vs `W_mature = max(...) = 30d`.** INCIDENT-MEMORY §5.3/§5.5/§7.2 matures a resolution at `applied_at + W_confirm`, states `W_confirm` is controller-owned with default `14d` and invariant `W_confirm ≥ W_recur`. TRUST-CONTROLLER §6.2 explicitly **"`W_confirm` removed"** and defers `matured` to `W_mature = max(W_recur, W_spawn, W_revert) = 30d`, giving a concrete race argument for why 14d is unsafe (a day-14 recurrence could be beaten to the record and banked good). So Incident Memory's maturation job fires at 14d and mints `confirmed_good` while `W_spawn`(14d, but off `applied_at`) and `W_revert`(30d) windows are still open — the exact boundary race TC says it closed. This is a **direct semantic contradiction on the attack-#8 defense**, not just a constant. **Keystone must delete `W_confirm` from Incident Memory and rebind its maturation job to `W_mature`;** TC is the correct authority and its argument is sound.

**3. The `applied` / `assisted_action` write path is unowned — L1→L2 bootstrapping has no producer.** TRUST-CONTROLLER §2.2 calls `assisted_action` (human-approved L1 merges written to the `auto_action` table with `applied_by='human_approved'`) "the single most load-bearing gap… the promotion ladder now has a data source." But: (a) INCIDENT-MEMORY §2 says `resolutions.auto_action_id` is set "IFF auto-applied… **NULL for … human-merged PRs**" and §7.1 emits events only for rows with `auto_action_id IS NOT NULL` — so Loop B *heals* (L1, human-merged) never get an action id and never accrue outcomes. This flatly contradicts TC's requirement that they do. (b) The `applied` event's producer is named as "Verification Gate on auto-apply, writing the `auto_action_id`" (INCIDENT-MEMORY §7.2) but VERIFICATION-GATE.md contains **zero** mentions of `auto_action`, `assisted_action`, `applied`, `ingestOutcome`, or `OutcomeEvent` — it emits only a stateless `GateResult`. **No component actually writes `auto_action` rows.** Keystone must assign: who inserts the `auto_action` row (both `machine` and `human_approved` variants), at what event (auto-apply commit; L1 PR merge), and reconcile Incident Memory's "NULL for human-merged" rule against TC's need for `assisted_action`s. Without this, TC can never leave L1 for any class — the entire promotion ladder is dead on arrival.

**4. `notify_state` is consumed but never owned or defined.** LOOP-A §8 makes delivery a durable **compare-and-set on `incident.notify_state`** (`investigating → notified`), the mechanism that guarantees no double-notify and race-safety with human resolution. No schema defines `notify_state`: INCIDENT-MEMORY's `incidents.status` enum is `open | diagnosed | resolved | closed` (a different field with different values), and the durable state machine that would own transitions lives in the unwritten `ORCHESTRATION.md`. **Keystone (ORCHESTRATION.md) must define `notify_state`, its column/store, its CAS semantics, and its relationship to `incidents.status`** — otherwise Loop A's central anti-double-notify guarantee has no backing store.

**5. `superseded_by_human` and `provisional_human_confirmed` are asserted labels that violate the schema CHECK constraint.** LOOP-A §6/§7 introduces retrieval-class `provisional_human_confirmed` and trace label `superseded_by_human`. INCIDENT-MEMORY §5.2 says `superseded_by_human` "maps to `wrong_rca`/neutral," but `resolutions.ck_outcome_label` permits only `proposed | applied | confirmed_good | recurred | reverted | wrong_rca | superseded`, and the §5.4 projection table has no row producing `provisional_human_confirmed`. Meanwhile LOOP-A §6 lists `provisional_human_confirmed` as a distinct retrieval polarity the caller acts on ("WEAK context only"). So Loop A believes a `provisional_human_confirmed` outcome exists and is retrievable; Incident Memory has no such stored value and no projection to produce it. **Keystone must reconcile the label vocabularies:** either add a stored representation for "human tapped confirm, watcher window open" (Loop A's whole §7 anti-rubber-stamp argument depends on it being distinguishable from `confirmed_good` and from `proposed`), or Loop A must stop treating it as a retrievable class. As written the two specs disagree on which labels exist.

## MAJOR — decisions/attacks double-owned, split-brained, or with a real seam

**6. The kill switch has three partial owners and no single mechanism-of-record.** SECURITY §7.1 says freeze is "enforced at the orchestrator… at every tier-transition (see ORCHESTRATION.md)" and is heartbeat-fail-safe. TRUST-CONTROLLER §5 forces every class's *effective level* to L0 on KILL and owns `earned_level`/`pre_kill_level`. VERIFICATION-GATE §7 independently checks a `frozen` flag and blocks its own downstream actions. LOOP-A/INCIDENT-MEMORY keep running under freeze. Each is locally coherent, but there is no single spec stating **where the authoritative kill bit lives, how the three enforcement points read it, and whether they can disagree** (e.g. gate sees `frozen=false` while controller has forced L0, or vice-versa during the heartbeat gap). This is exactly the split-brain VERIFICATION-GATE warns about for tiers. **ORCHESTRATION.md must own the kill bit and define how TC's `effectiveLevel`, the gate's `frozen`, and the orchestrator's tier-transition guard all derive from it.**

**7. `effectiveLevel` → gate `(loop, tier)` handoff is unowned, and the level/tier vocabularies don't line up.** VERIFICATION-GATE §1/§5 is handed an "already-resolved `(loop, tier)`" plus `requiredMutationScore` and `accountabilityOwner` by TRUST-CONTROLLER. But TC's interface returns `effectiveLevel(...) → AutonomyLevel (0..3)`, and never mentions producing `tier (1..4)`, `requiredMutationScore`, or `accountabilityOwner` in its output. The gate's tier is `1|2|3|4`; the controller's level is `L0..L3`; ARCHITECTURE-ORIGINAL's tiers are `1..4`. **Three numbering systems** (Tier 1–4, L0–L3, loop A/B/C) are used interchangeably across specs with no crosswalk. Who maps `AutonomyLevel → (loop, tier, requiredMutationScore, accountabilityOwner)` and calls the gate? That router is ARCHITECTURE-ORIGINAL §5's "Risk Classifier," which no reframed spec owns. **Keystone (ARCHITECTURE-REFRAMED.md + ORCHESTRATION.md) must define the router, the L↔tier crosswalk, and where `requiredMutationScore`/`accountabilityOwner` are sourced before the gate is called.**

**8. `assisted_action` outcome attribution requires spawn/recurrence/revert detection on L1 human-merged diffs, but Incident Memory's detectors are wired only to auto-actions.** Beyond the write-path gap (#3), even if `assisted_action` rows existed, INCIDENT-MEMORY §7.2's detector table attributes `spawn`/`recurrence`/`revert` and the §7.3 harm query all filter `auto_action_id IS NOT NULL`. TC's `l1_merged_harm_rate` (§6.1) and condition-2 promotion gate *depend* on caused-incident attribution for `human_approved` actions. **Keystone must extend Incident Memory's attribution + harm query to cover both `applied_by` variants**, or TC's L1-merged-harm safety number (the "rubber-stamp band" defense, its stated closure of attack #3) is uncomputable.

**9. D9 accountability owner: two components claim to *enforce* it, neither is wired to the other.** VERIFICATION-GATE §5 hard-FAILs the auto column if `accountabilityOwner` is null "in its input" — but the input comes from TC, and TC's `effectiveLevel` output doesn't carry it (#7). TRUST-CONTROLLER §3.6 enforces owner-non-null via a `trust_class` CHECK constraint at L2+. INCIDENT-MEMORY freezes `resolutions.merged_by` as "the D9 accountability owner." SECURITY §6 defines `accountability.yaml` with `owner_field` per loop. That's **four representations of the owner** (`trust_class.owner`, gate input `accountabilityOwner`, `resolutions.merged_by`, `accountability.yaml`) with no stated single source. **Keystone must declare `trust_class.owner` (or `accountability.yaml`) authoritative and define the propagation path into the gate input and into `resolutions.merged_by`.**

**10. Attack #11 / diff-stacking: closed by TC, but TC declares it does NOT exist as a stress-test finding — and two specs cite it as if it does.** TRUST-CONTROLLER §4.1 and D10-INSTRUMENT explicitly state STRESS-TEST has only attacks #1–#9 and label the churn escalator a "net-new guard." VERIFICATION-GATE (§1, §5, Attacks section) repeatedly cites "**attack #11 (diff-stacking)**" as a named stress-test finding it defers to TC §4.1. LOOP-A/INCIDENT-MEMORY don't use the number. The task prompt itself references "#11." **This is a coherence defect: the guard is real and correctly owned by TC §4.1, but the *numbering* is inconsistent** — VERIFICATION-GATE invents a stress-test "#11" that the authoritative TC spec says does not exist. Keystone must standardize: either (a) formally register diff-stacking as a net-new attack with a stable id and update all specs, or (b) strip the "#11" label from VERIFICATION-GATE and call it the net-new churn guard as TC does. The *defense* lines up (gate emits `moduleArea`/`diffLines`; TC §4.1 consumes them); only the name is incoherent.

**11. Gate ↔ churn-counter feedback loop has a data-source seam.** VERIFICATION-GATE emits `moduleArea`/`diffLines` on `GateResult` "consumed by TRUST-CONTROLLER §4.1 churn." But TC §4.1 says the churn counter is "derived deterministically from the `auto_action` table windowed by `applied_at`" — i.e. from *applied actions*, not from *gate results*. A gate PASS that never becomes an applied action (e.g. Loop C PR awaiting HITL, or a heal not yet merged) is a proposed change to the area but is not in `auto_action`. So the two specs describe **two different churn inputs** (gate-result stream vs applied-action table). **Keystone must state which stream feeds churn** — and if it's `auto_action`, then VERIFICATION-GATE's claim that its `moduleArea`/`diffLines` fields feed the churn escalator is misleading and should point at the apply-time writer instead.

## MINOR — seams the keystone should still note

**12. `module_area` depth default is stated three times; keep it single-sourced.** VERIFICATION-GATE §2.0, TRUST-CONTROLLER §2.1, and INCIDENT-MEMORY §2 all independently define `module_area` = repo dir at "fixed depth (default 2)." They agree today, but three copies of a load-bearing key definition drift. Keystone should name one owner (TC, since it keys autonomy on it) and have the others reference it.

**13. `mutation_score` / `requiredMutationScore` floor lives in two config files.** VERIFICATION-GATE's `verification.yaml` holds `mutation.floor`; the *effective* bar comes from TC at call time. But TC's config (`trust_controller.yaml`) never mentions a mutation bar, and §3.5 defers it to "policy floor… checked by the Verification Gate." So the "effective `requiredMutationScore`" the gate expects from TC has no defined source in TC's config. Minor because the gate has a hard floor fallback, but keystone should say where the per-class effective bar is stored.

**14. Loop A's gate config knobs claim shared ownership that isn't reflected.** LOOP-A §4 says "VERIFICATION-GATE.md owns the shared knobs" for `grounded_confidence` (`g2_sample_size`, etc.). VERIFICATION-GATE.md's `verification.yaml` contains no `grounded_confidence` block — those are Loop-A-only RCA knobs, unrelated to the gate's signal battery. Minor mislabel; keystone should note Loop A owns its own gate config (the gate proper never sees a WhyTrace).

**15. `ORCHESTRATION.md` and `HITL-APPROVAL.md` are referenced by 4 and 3 specs respectively but do not exist.** ORCHESTRATION.md is load-bearing for: `notify_state` (#4), kill-switch enforcement (#6), the router/level→tier mapping (#7), the durable state machine (D1), and who invokes the gate and Loop A. HITL-APPROVAL.md owns: the async approval ladder (ARCHITECTURE-ORIGINAL §8, STRESS-TEST attack #6), the Telegram bot Loop A/TC/gate all ping, kill-switch release auth (SECURITY §7.1). **These are the two missing keystones; most BLOCKERs above land in ORCHESTRATION.md.**

## What the keystones must reconcile (summary for the writer)

1. **ORCHESTRATION.md** — resolve BLOCKERs #4 (notify_state), and MAJORs #6 (kill bit single source), #7 (router + L↔tier crosswalk + gate-input sourcing). Define who invokes Loop A and the gate.

Hold the shared window/harm-metric definitions and the build-order that D10 gates.
3. **HITL-APPROVAL.md** — own attack #6 (the still-open off-hours autonomy-vs-trust inversion for L2), and the shared approval/kill-release channel.
4. **Numbering hygiene (#10, #12, #13):** one crosswalk for Tier 1–4 / L0–L3 / loop-A/B/C; strip or formalize "attack #11"; single-source `module_area` and the mutation floor.

The six specs are individually strong and their *defenses* line up; the failures are at the **seams the two unwritten keystones are supposed to own** — every BLOCKER is a contract between two components that each assumed the other (or a keystone) would define the joining field.
