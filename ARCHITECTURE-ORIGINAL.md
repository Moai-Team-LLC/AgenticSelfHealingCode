# Self-Healing Ops: сеть агентов для мониторинга, ремонта и тестового покрытия продукта

> Исходный target-state документ пользователя. Сохранён как источник. Критический
> разбор — в STRESS-TEST.md; переработанная архитектура — в ARCHITECTURE-REFRAMED.md.

Детальная архитектура полного решения. Не MVP — целевое состояние системы, с указанием, какие узлы можно строить поверх готовых инструментов, а какие писать самому.

---

## 0. Принципы
1. **Автономия — спектр, не переключатель.** Каждое действие классифицируется по риску (Tier 1–4), а не проходит через единый гейт «approve/reject». Иначе либо ревьюер начинает штамповать «да» не глядя, либо автономию вообще не включают.
2. **Ничего не мержится без зелёных тестов**, и каждый фикс обязан либо добавить регрессионный тест, либо явно расширить покрытие в затронутой области.
3. **Всё replayable.** Каждый инцидент — от сигнала до мержа — это трасса (why-trace), которую можно воспроизвести и по которой можно судить систему постфактум, а не просто текстовый лог.
4. **Доверие наращивается по данным.** Автономия каждого класса действий расширяется на основе измеренного human override rate, а не задаётся один раз и забывается.
5. **Есть kill switch.** Одна команда — вся сеть переходит в режим «только диагностика», без единого автоприменённого изменения.
6. **Garbage in, garbage out.** Если существующий тестовый набор слабый, автомерж Tier 2 держим выключенным до тех пор, пока coverage-агент не поднимет покрытие критичных путей — это буквально bootstrapping-проблема, и её нельзя пропускать.

## 1. Топология пайплайна
```
Signal Layer (backend + frontend + infra)
        │
        ▼
Aggregation & Dedup Agent  ───────►  Incident Memory (pgvector)
        │                                     ▲
        ▼                                     │ (retrieval для контекста)
Triage / RCA Agent  ◄────────────────────────┘
        │
        ▼
Risk Classifier ──────────────► Tier 1 (read-only) → в память, без действий
        │
        ├── Tier 2 (reversible, low blast radius)
        ├── Tier 3 (external-facing / умеренный риск)
        └── Tier 4 (high-risk / необратимо)
        │
        ▼
Repair Worker (backend-fix-agent | frontend-fix-agent)
        │
        ▼
Verification Gate (тесты, типы, линт, судья-агент)
        │
        ▼
HITL Approval Layer (Tier 3–4 всегда; Tier 2 — auto с логом)
        │
        ▼
Merge & Deploy (canary / feature-flag wrap)
        │
        ▼
Incident Memory (запись исхода) ──► Alerting Layer

Параллельно, отдельным расписанием:
Coverage Agent (backend + frontend) → PR с новыми тестами → тот же Verification Gate
```
Всё это работает под управлением **Durable Orchestrator** — координатора, который переживает рестарты и держит состояние инцидента между «агент работает» и «ждём человека 40 минут» (см. раздел 9).

## 2. Слой сигналов (Observability)
Не изобретай велосипед на этом слое — используй существующие источники, а сеть агентов вешай поверх через webhooks/API.

**Backend:** структурированные логи, OpenTelemetry-трейсы, метрики (латентность, error rate, saturation), события деплоев и feature-flag изменений, события инфраструктуры (k8s restarts, OOM kills). Деплой-события критичны для RCA — большая часть инцидентов коррелирует с последним изменением, и агент должен это видеть первым делом, а не искать вслепую.

**Frontend:** JS-ошибки, unhandled promise rejections, Core Web Vitals, session replay, сетевые ошибки API-вызовов из браузера, console errors. Здесь либо Sentry browser SDK, либо самописный RUM-коллектор поверх `window.onerror`/`PerformanceObserver`.

**Business-сигналы:** падение конверсии на шаге воронки, рост отказов оплаты, аномалии в бизнес-метриках — часто это первый сигнал о проблеме, которую error tracking ещё не увидел (например, тихая деградация без исключений).

Всё нормализуется в единый контракт **Incident Candidate**:
```json
{
  "id": "uuid",
  "source": "sentry | otel | rum | business-metric",
  "fingerprint": "hash по стеку/локации",
  "severity": "raw score",
  "first_seen": "ts",
  "occurrences": 0,
  "affected_service": "string",
  "affected_paths": ["repo/path/to/file.ts"],
  "recent_deploys": ["deploy_id, ts, diff_url"],
  "raw_payload": {}
}
```

## 3. Aggregation & Dedup Agent
Группирует сырые события в инциденты по fingerprint (тот же принцип, что issue grouping в Sentry), подавляет шум порогом (N событий за окно), считает приоритет как `blast_radius × frequency × business_criticality`. Это лёгкий, преимущественно детерминированный слой — LLM здесь почти не нужен, только на этапе классификации новых, ещё не виденных паттернов ошибок.

---

## 4. Триаж-агент (Root Cause Analysis)
Самый сложный и самый ценный узел сети. Это investigation-агент, а не rule-based alerting: он сам решает, какие инструменты запускать и в каком порядке.

**Инструментарий агента:**
- Полнотекстовый и семантический поиск по кодовой базе
- `git blame` / `git log` по затронутым файлам, диффы последних деплоев
- Trace waterfall и корреляция логов по trace_id
- Ретрив похожих прошлых инцидентов из Incident Memory (few-shot контекст: «мы это уже чинили в марте, вот как»)
- Опционально — выполнение кода в изолированной среде для воспроизведения

**Выход агента — структурированный RCA-отчёт:**
- гипотеза первопричины + confidence score
- затронутые компоненты
- класс фикса (код / конфиг / инфра / данные)
- предполагаемый risk tier
- альтернативные гипотезы, если confidence ниже порога

Если RCA не сходится выше порога уверенности — агент **не пытается чинить вслепую**, а сразу эскалирует человеку с текущими находками. Это осознанный отказ от «всегда что-то предложить»: уверенность LLM систематически завышена, и при многошаговой цепочке (RCA → фикс → верификация) ошибка калибровки компаундится — три шага с честными 90% на каждом дают на выходе около 42% вероятности, что вся цепочка верна. Порог отсечения на триаже — самая дешёвая точка, чтобы это остановить.

## 5. Risk Classifier и политика маршрутизации
Четырёхуровневая модель — общепринятый паттерн для production-агентов с write-доступом:

| Tier | Тип действия | Пример | Маршрут |
|---|---|---|---|
| 1 | Read-only | диагностика, RCA без изменений | автономно, лог |
| 2 | Reversible, узкий blast radius | null-check, off-by-one, изолированная функция с полным покрытием | auto-apply при зелёных тестах + путь не в protected-листе |
| 3 | External-facing / умеренный риск | API-контракт, UI-компонент, изменение поведения | PR обязателен, мерж только человеком |
| 4 | High-risk / необратимо | миграции схемы, auth, billing, инфра, секреты | синхронное согласование до любых изменений кода, агент только предлагает план |

Политика задаётся конфигом, а не зашита в промпт:
```yaml
risk_policy:
  protected_paths:
    - "src/auth/**"
    - "src/billing/**"
    - "infra/**"
    - "**/migrations/**"
  tier2_max_diff_lines: 15
  tier2_requires:
    - all_tests_green
    - coverage_delta_non_negative
    - no_new_dependencies
  escalation:
    tier3_channel: "telegram:#repairs-review"
    tier4_channel: "telegram:#repairs-critical"
    tier4_sync: true
```
Диффы, которые превышают ожидаемый размер для класса бага, автоматически поднимаются на tier выше — независимо от исходной классификации.

## 6. Repair Workers
Два специализированных субагента, а не один универсальный — у фронтенда и бэкенда разные конвенции, тест-раннеры и линтеры:
- **backend-fix-agent** — доступ к серверному коду, БД-схемам (только чтение схемы, не данных), интеграционным тестам
- **frontend-fix-agent** — доступ к UI-коду, снапшот-тестам, визуальной регрессии

**Исполнение:** эфемерный sandboxed контейнер на задачу, чистый клон репозитория, без доступа к прод-секретам — только service account с минимальным scope. Разрешённые действия: чтение/запись в репо, прогон тестов и линтера, git-операции без force push и без прямого push в main. Запрещено: команды, меняющие инфраструктуру, доступ к секретам, доступ к прод-БД.

**Цикл:** воспроизвести → патч → прогнать тесты локально → self-review диффа → повторить до зелёного или до бюджета (лимит итераций/токенов). Если не сходится за N попыток — агент останавливается и эскалирует человеку с частичной работой и объяснением, что не получилось, вместо бесконечного зацикливания.

---

## 7. Verification Gate
До того как PR вообще увидит человека — автоматический слой проверки:
- полный тест-сьют (не только новый/изменённый тест)
- типы, линт, статический анализ
- **судья-агент** — отдельный LLM-проход, который читает диф и ищет scope creep, хардкод секретов, очевидно небезопасные паттерны, несоответствие RCA заявленному фиксу

Это прямая параллель judge-gated self-improvement — независимая проверка результата отдельным процессом, а не тем же агентом, который писал код.

---

## 8. HITL Approval Layer
**Канал:** Telegram/Slack-бот с инлайн-кнопками approve/reject/edit и deep-link на полный диф + RCA в веб-интерфейсе. Контекст, который видит человек при согласовании:
- сжатая RCA (не сырые логи)
- confidence score
- диф
- результаты тестов
- оценка blast radius
- похожие прошлые инциденты и их исход (из Incident Memory)

**Асинхронно по умолчанию.** Синхронная блокировка ломается на реальной инфраструктуре (таймауты, протухшие токены) — вместо этого очередь с таймаутом и лестницей эскалации: нет ответа за N минут → вторичный аппрувер (для Tier 3) или автоотказ с уведомлением (для Tier 4, где действие не должно проходить по умолчанию).

**Обратная связь в политику:** override rate трекается по категориям. Если люди систематически отклоняют определённый класс Tier 2 автопримененных фиксов — это не «ревьюер придирается», это сигнал сузить политику для этого класса.

## 9. Test Coverage — два разных механизма
Их часто путают, но это разные агенты с разным ритмом.

### 9.1. Регрессионный тест при фиксе (обязательный гейт)
Repair worker не может закрыть задачу без теста, воспроизводящего исходный баг. Это часть Verification Gate из раздела 7, не отдельная фича.

### 9.2. Coverage-agent (по расписанию, отдельная лента PR)
Сканирует изменённые/непокрытые пути по coverage-отчётам, приоритизирует по трафику и риску (а не просто по проценту строк), генерирует тесты, открывает отдельный PR с более низким приоритетом ревью.

**Backend:** unit + integration + contract-тесты на границах API — контрактные тесты особенно важны, если сервисы независимо деплоятся.
**Frontend:** component/unit-тесты + визуальная регрессия (скриншот-диффинг) + E2E на критичных пользовательских путях (Playwright).

**Важная калибровка, которую стоит знать заранее:** генерация новых тестов — не то, что даёт наибольший ROI. По опыту продакшен-внедрений, наибольшую ценность даёт **самопочинка существующих тестов** — починка тестов, сломанных легитимным изменением кода, и обнаружение/карантин флаки-тестов, а не наращивание количества новых. Если бюджет на coverage-agent ограничен, приоритет: сначала self-healing существующего сьюта, потом закрытие дыр в покрытии.

## 10. Incident Memory (why-trace хранилище)
Здесь прямая переиспользуемая инфраструктура с AgenticMind — тот же Postgres + pgvector, тот же принцип replayable why-trace и judge-gated проверки, только применённый к операционным инцидентам, а не к ответам с цитированием.

Схема хранит: сигнал → RCA → путь решения → фикс → исход → фидбек человека, всё эмбеддится для семантического ретрива. Каждый новый инцидент на этапе триажа подтягивает похожие прошлые — это не просто аудит-лог, а рабочий контекст, который не даёт агентам дважды наступать на одни и те же грабли одним и тем же неправильным способом.

---

## 11. Durable Orchestrator
Жизненный цикл инцидента может растянуться на часы (ожидание человека) и должен переживать рестарты процесса. Разделение ответственности:
- **Исполнение задачи агента** — Claude Agent SDK (та же архитектура, что у Claude Code: hooks, MCP, subagents из коробки, TS-native), запускается на каждый шаг пайплайна
- **Долгоживущее состояние поверх него** — durable-слой с чекпоинтингом и нативным HITL-прерыванием (`interrupt`), который держит состояние между «агент работает» и «ждём approval». Для TS/Postgres-стека это либо LangGraph.js, либо своя стейт-машина на Postgres, если не хочется тащить отдельный рантайм. Temporal — вариант, если нужна инфраструктурная надёжность в духе финансовых систем, ценой более тяжёлой операционки.

Claude Agent SDK сам по себе не абсорбирует durability между рестартами процесса — поэтому пара «SDK для исполнения + durable-слой для координации» — рабочее разделение, а не дублирование.

## 12. Guardrails (сквозные, применяются на каждом узле)
- Заблокированные пути и команды: секреты, CI-конфиг, infra-as-code, миграции — никогда не автономно
- Rate limits / circuit breaker: максимум фиксов в час, автопауза пайплайна при повторных фейлах или повторных отказах человека
- Полный неизменяемый аудит-лог каждого решения и вызова инструмента
- Обязательный rollback-план для каждого автопримененного Tier 2 фикса (feature-flag обёртка или готовый revert-коммит рядом с фиксом)
- Kill switch — одна команда замораживает всю автономию, оставляя только диагностику

---

## 13. Метрики
- MTTR до/после по каждому классу инцидентов
- Escalation rate и human override rate по тирам
- Точность RCA (подтверждённая человеком причина совпала с гипотезой агента)
- Success rate фикса с первой попытки
- Coverage delta во времени, отдельно frontend/backend
- Стоимость LLM на инцидент

---

## 14. Поэтапный ввод в эксплуатацию
Даже для целевой архитектуры — вводить autonomy тиры нужно по фактическим данным, а не сразу целиком:
1. **Фаза 1** — только детекция + RCA, ноль автоприменённых изменений. Цель: набрать статистику точности RCA.
2. **Фаза 2** — Tier 1–2 автоприменение за feature-флагом/канарейкой, Tier 3–4 только как PR. Coverage-agent ещё выключен.
3. **Фаза 3** — расширение автотиров по измеренному override rate, включение coverage-agent.
4. **Фаза 4** — межсервисный охват, предиктивная детекция (до того как сломается для пользователя), а не только реактивная.

---

## 15. Где предел автономии
Система не заменяет человека там, где:
- проблема требует архитектурного решения, а не патча
- вопрос про capacity/scaling planning
- инцидент — это security incident, требующий немедленного человеческого суждения, а не автопатча
- логика фикса зависит от неоднозначного продуктового/бизнес-решения, которое не выведешь из кода
- существующий тест-сьют слишком слабый, чтобы верификация вообще что-то значила (см. принцип 6)

## 16. Сводный стек
| Слой | Рекомендация |
|---|---|
| Телеметрия | Sentry (или self-hosted OTel-коллектор) + browser SDK для RUM |
| Исполнение агентов | Claude Agent SDK (TS) |
| Durable orchestration | LangGraph.js / своя стейт-машина на Postgres, либо Temporal для тяжёлого варианта |
| Incident memory | Postgres + pgvector (переиспользуется инфра AgenticMind) |
| Backend testing | unit + integration + contract, coverage-репорты как вход coverage-agent |
| Frontend testing | Playwright (E2E + визуальная регрессия), component-тесты |
| HITL-канал | Telegram/Slack-бот с инлайн-approve + веб-UI для полного диффа |
| CI/CD интеграция | GitHub Actions / GitLab CI как триггер и как sandbox для repair worker |
